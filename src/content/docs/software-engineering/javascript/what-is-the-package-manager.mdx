## 패키지 매니저란?

- `JavaScript`나 `TypeScript`를 사용하다보면, `require`나 `import` 구문을 통해 외부 의존성을 참조하게 됨.
- 이러한 외부 의존성을 올바르게 참조할 수 있도록 도와주는 프로그램이라 생각하면 됨.

```js
import React from '/Users/raon0211/path/to/react/index.js';
import { sum } from '/Users/raon0211/path/to/@toss/utils/index.js'
```

- JavScript 표준인 ECAMScript에 따르면, 원래는 정확한 절대경로나 상대 경로를 통해서만 `import`할 수 있음.
    - 실제로 Deno나 브라우저의 JavaScript 표준 문법을 보면 정확한 절대 겨올를 사용하고 있음.
- 대부분의 사람들은 아래와 같은 코드의 형식으로 사용하고 싶어함.

```js
import React from 'react';
import { sum } from '@toss/utils';

sum(1, 2, 3);
```

```js
const _ = require('lodash');
```

- 하지만, 위와 같은 코드의 형식으로 사용하게 된다면, 문제가 발생할 수 있음.
    - 예를 들어, `react`를 사용하고 싶다면, 정확히 어떤 버전인지 모호하다는 문제가 발생할 수 있음.
    - React는 18.0.1일 수도 있고, 최근에 나온 버전의 베타일 수 있도 있음.

- 이러한 정확한 패키지 정보는 ```package.json``` 파일에 명시되어 있음.

```json
{
  "dependencies": {
    "react": "^18.2.0" // react는 ≥ 18.2.0, <19 사이의 어떤 버전이든지 쓸 수 있다고 명시
  }
}
```

- `package.json` 파일에 명시된 의존성 정보를 바탕으로, 모든 소스 코드 파일이 특정 버전의 라이브러리를 사용할 수 있도록 보장함.
- 일반적으로 `package.json` 파일에 디펜던시를 명시하고, `npm install` 혹은 `yarn install`을 하면, 해당 의존성의 명시된 버전을 설치하게 됨.
- **즉, 패키지 매니저가 앞서 이야기 한 모호한 버저닝 문제를 해결 줌.**

### 패키지 매니저가 동작하는 세 단계

-  Yarn을 포함한 패키지는 일반적으로 세 단계로 동작함. `Resolution`, `Fetch`, `Link` 단계로 나뉨.

#### Resolution 단계

- 패키지 매니저는 Resolution 단계에서, `package.json` 파일에 명시된 버전 범위에 따라 정확한 버전을 결정함.
    - 예를 들어, “react: ^18.2.0” 이라고 명시되어 있으면, 유의적 버전 규칙에 따라 `≥ 18.2.0, < 19`사이의 어떤 버전이든 사용할 수 있음.
    - 패키지 매니저는 이러한 범위를 만족하는 선에서 가능한 최신 버전을 사용하려고 함. **즉, 최신 버진인 18.3.1을 선택할 수 있음.**
- 설치한 라이브러리가 사용하는 다른 라이브러리, 즉 의존성의 의존성 문제가 있음.
    - JavaScript에서는 패키지끼리 의존성을 갖는 상황이 흔함. 예를 들어 `@toss/use-overlay`는 `react`를 사용함.
    - 그런데 `react`도 의존성을 가지고 있음. 의존성이 또 어떠한 의존성을 가지고 있는지 확인하는 작업이 필요함.
- 그 의존성의 버전도 고정해야함. JavaScript에서는 위와 같은 의존성의 버전을 범위로 명시하고, 패키지 간에도 의존성을 가지기 때문에, 똑같은 `package.json`에 대해서도 사용하는 의존성 버전이 완전히 달라질 수도 있음.
- 정리하면, Resolution 단계는 모든 기기에서 고정된 버전을 사용할 수 있도록 함. 의존성 버전을 전부 고정시키고, 의존성의 의존성을 다 찾아서 그 버전도 고정시키며, 그러한 결과물을 `yarn.lock` 혹은 `package-lock.json` 파일에 저장함.

#### Fetch 단계

- Resolution 단계에서의 결과로 결정된 버전을 실제로 다운로드하는 과정임.
    - `yarn.lock`에 명시된 패키지가 있음. 패키지 매니저는 이 과정에서 네트워크를 통해 필요한 파일들을 가져옴.
- 이러한 과정에서 네트워크를 통해 필요한 파일들을 가져오는 과정이 Fetch 단계임. 일반적으로, 99%는 npm 레지스트리에서 받아옴.

#### Link 단계

- 가장 까다로운 Link 단계임. npm, pnpm, PnP 사례를 각각 살펴봄.

##### [1] npm Linker

- 가장 익숙한 `node_modules` 기반의 Linker를 살펴봄. `package.json`에서 명시하는 모든 의존성을 `node_modules` 디렉토리 밑에다가 하나하나식 쓰는 방식이 npm Linker의 역할임.
- 예를 들어 소스 코드에서 React와 TDS 모바일 라이브러리를 사용한다면, `my-service`의 `node_modules` 하위에 React랑 TDS 모바일 패키지를 추가함.
- TDS 모바일 패키지에도 `node_modules`가 있다면 `@radix-ui/dialog` 패키지를 또 그 밑에 깔아주는 것이 npm Linker의 역할임.

```
my-service/
└─ node_modules/
|  ├─ react/
|  |  
|  └─ @tossteam/tds-mobile/
|     └─ node_modules/
|         └─ @radix-ui/react-dialog
└─ src
    └─ index.ts
```

- 패키지를 찾으려면 `node_modules` 디렉토리를 계속 타고 올라가면서 파일을 여러 번 읽게 됨.
    - `import`나 `require` 하는 속도가 느려지게 됨. 또한 디렉토리 크기가 너무 커짐.
    - 실제로 파일 시스템에 디렉토리와 파일을 하나하나 만들고 쓰기 때문임.
- 예를 들어 만약 100개 프로젝트에서 React 18.2.0 버전을 쓴다고 하면 정말로 100번씩 React 18.2.0이 추가됨.
    - [호이스팅(Hosting)](https://toss.tech/article/node-modules-and-yarn-berry)
